# TrendScope MVP Plan

> 版本：v1.0
> 日期：2026-03-17
> 状态：执行建议

## 1. 目标

在 `3 周` 内交付一个可自用、可部署、可演示的 MVP，范围严格限定为：

- 搜索 GitHub 项目或普通关键词
- 返回趋势图、快照、内容列表
- 支持加入追踪和定时补数

不以“多数据源铺满”作为阶段目标。

## 2. 里程碑

### M1. 基础链路打通

目标：把“关键词 -> 数据入库 -> API 出数”跑通。

工期：第 1 周

工作项：

- 固定技术栈和目录结构
- 建表和迁移
- 实现关键词规范化
- 实现 GitHub 历史采集
- 实现 NewsNow 快照采集
- 实现 `GET /api/search`

完成标准：

- 空库启动后可搜索 `owner/repo`
- 搜索能写入 `keywords`、`trend_points`、`backfill_jobs`
- 可以从 API 读回基础结果

### M2. 搜索结果页与异步回填

目标：让用户能真正使用搜索页。

工期：第 2 周

工作项：

- 前端搜索页骨架
- 趋势图组件
- 今日快照卡片
- 内容列表
- 回填状态轮询接口
- GitHub / NewsNow 部分成功渲染
- 加入追踪按钮

完成标准：

- GitHub 项目首次搜索能看到历史曲线
- 普通关键词首次搜索能看到当前快照和内容列表
- 任一来源失败时页面仍可用

### M3. 稳定性与部署

目标：把 demo 提升到可复现交付。

工期：第 3 周

工作项：

- 定时采集任务
- 错误日志与重试
- Dockerfile / docker-compose
- `.env.example`
- README 启动文档
- 基础测试和验收脚本

完成标准：

- 新环境按文档可启动
- 已追踪关键词能自动补数
- 关键接口和采集链路有最小回归测试

## 3. 依赖顺序

必须按以下顺序推进：

1. 先定数据模型
2. 再做采集和回填任务
3. 再做 API 聚合
4. 最后做页面组合和交互细节

不要在数据链路未稳定前提前做：

- 综合指数
- 权重面板
- 发现页
- 推送系统

## 4. 人日估算

| 模块 | 预估 |
|---|---|
| 数据模型与迁移 | 1.5 天 |
| GitHub 采集与回填 | 2 天 |
| NewsNow 采集 | 1.5 天 |
| 搜索服务与任务状态 | 2 天 |
| 搜索结果页前端 | 3 天 |
| 定时任务与日志 | 1.5 天 |
| Docker 与文档 | 1 天 |
| 测试与修复缓冲 | 2 天 |

总计约 `14.5` 人日，留出约 `0.5-1` 周缓冲更合理，因此建议按 `3 周` 计划执行，而不是原稿中的 `2 周`。

## 5. 测试门槛

### 5.1 后端

- 输入分类单元测试
- GitHub URL 规范化测试
- 回填任务状态流转测试
- 外部 API 失败时的部分成功测试

### 5.2 前端

- 搜索页首次加载
- 回填轮询后的图表更新
- 来源失败时的灰态提示
- 加入追踪状态切换

### 5.3 集成

- 空库启动到首次搜索成功
- GitHub 项目冷启动成功
- 普通关键词冷启动成功
- 定时任务对已追踪关键词生效

## 6. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| NewsNow 公网限流或不稳定 | 搜索快照不完整 | 保留自部署切换位，页面支持 partial |
| GitHub API 配额不足 | 仓库历史无法补全 | 强制使用 Token，做最小缓存 |
| 普通关键词缺乏历史数据 | 图表信息量不足 | 明确展示“从开始追踪起积累” |
| 需求继续膨胀 | 排期失控 | 任何新增功能必须先判断是否属于 MVP 非目标 |

## 7. 首发后再做

以下项目只能在 M3 验收后重新排期：

- 综合指数
- 权重预设
- 发现页
- Google Trends
- 导出
- 事件标注
- 推送通知
